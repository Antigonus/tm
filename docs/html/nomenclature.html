Nomenclature and Conventions

A myriad of terms now exist in computer science for talking about memory and end cases, but still some things have no conventional terms. Hence, I feel it is important to provide a consistent nomenclature for use when talking about the tape machine library.

A tape machine has a tape, an indicator head (or just head), and a function interface.

The tape consists of an array of one or more cell(s). Each cell contains exactly one object. An object is either a) a subspace or b) a data blob that one hopes has some meaning for the tape machine user. Spaces are discussed in the chapter "Spaces".

Because cell allocation requires an initialization object, it is not possible to have an uninitialized cell. If a machine can not be initialized, or it must become empty, the machine in its entirety may be replaced by ∅ (nil). When no initialization value is provided for making a tape, we build a one cell machine, where the object in that cell is a symbol representing the type of the tape. When building macros this is actually what is needed. For other cases there is a library function, d◧ (deallocate leftmost), which will drop the first cell from a tape.

When a tape machine tape has two or more cells, all cells, potentially except for two, have two neighbors. A left neighbor and a right neighbor. The two potentially special cells are called leftmost, which has only a right neighbor, and rightmost, which has only a left neighbor. The tape may be an abstraction that is maintained through function calls, in which case the tape might not have a leftmost or rightmost.

The leftmost object is the object in the leftmost cell, etc.

A one cell macchine represents a singularity in our system, as the one cell might be defined to be any combination of being its own left neighbor, being its own right neighbor, being leftmost and/or being rightmost. We provide two such examples in the library, tm-singular-affine, where a one cell machine tape has no leftmost or rightmost, and the one cell is both its own left neighbor and right neighbor, and tm-singular-projective where the one cell is both rightmost and leftmost.

At any given time the indicator head is on exactly one cell. We say indicator head, rather than the more conventional read-write head, because reading and writing are part of the function interface. The purpose of the indicator head is to locate a cell. The object in the indicated cell is called the indicated object, or the object indicated by the head.

Putting the head on a specific cell is to cue the head. Three cue functions are defined in the library: cue-leftmost, cue-to, and cue-rightmost. Cue-to cues the head to the same tape and location as for another tape machine. A related function dup both makes a new tape machine and cues it to another machine. With these functions it is possible to put more than one head on the same tape. Having multiple heads on one tape leads to potential hazards when combined with structural operations. This is discussed in the chapter, "Issues".

Heads are either stepped or cued. We do not apply other motion verbs to heads. In contrast, objects are said to be moved. A head is not a tape machine, but it is unambiguous to say "step a tape machine". This actually means to step the tape machine's head. However, if one says that "a tape machine moved", that means the tape machine is an object and it was moved to another cell. This distinction between the connotations of stepping and moving becomes particularly important after the introduction of "spaces".

The function interface consists of primitive and derived functions. Primitive functions are those with no generic implementation. A new tape machine may be defined by providing an implementation for the primitive functions, and that would be sufficient as the derived functions have generic implementations in terms of the primitive functions. There are many derived functions including those for handling subspaces, for quantification and repetition, and for compound operations.

In this implementation, tape-machine is a class with two slots. One slot is called, HA which stands for head address, and the other slot is called tape. Should the tape machine be mounted on a sequence, the tape slot is set to the sequence, and HA holds the location in the sequence. For an array HA will be the index for the current cell. For a list HA will be the current cons cell. Derived classes make do with just these two slots, which assures that we can consistently cue-to back and forth between tape-machine types.

We have two functions for making tape machines, one is tm-mk and the other is mount. The function mount takes only a sequence and then tries to match that up with the best tape machine implementation for the type of sequence. Where as tm-mk accepts a tape machine class and an initialization value, and then tries to produce an initialized tape machine of the specified class type. The function unmount accepts a tape machine and returns the mounted tape.

Modifications to head location or to the tape contents are said to be stateful. Modifications that add or remove cells from the tape are said to be structural. (Adding cells to the tape is called allocation. Removing cells from the tape is called deallocation.) We know from experience that programs that only modify state tend to be simpler than those that modify structure. Yet there always seem to be cases, such as gathering results, where structural changes are the most convenient type. Even Scheme allows one to use a cons cell to lengthen a list, albeit only from one end.

We use inclusive bounds so we don't need to represent addresses that are outside of our address space. The typical mode is to do some work, and then to step. If the machine head is on rightmost, then we do something appropriate.
